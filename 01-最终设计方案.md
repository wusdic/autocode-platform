# 多项目自动编程平台 · 最终完整设计方案

> 基于 Hermes Agent **v0.16.0 (v2026.6.5, 2026-06-05)** 官方文档逐项验证
> 配套文档：《02-从零开始操作手册.md》（可执行的 step-by-step）

---

## 0. 设计总纲（一页看懂）

**一句话**：每个项目 = 一个独立 Hermes 实例 + 一块独立 Kanban 看板；项目内由 CEO（只沟通、不干活）通过看板把任务派给一组"命名角色 profile"（产品/架构/研发/质控/变更管控），各角色用不同大模型协作与互相质疑；用户只通过你自建的 Web/API 网关对接 CEO，其余全自动。

**五条不可动摇的底层原则**（每条都有官方依据，见附录）：

1. **隔离靠基础设施，不靠提示词**。profile 只隔离记忆/配置，**不是沙箱**；执行必须跑 Docker backend + 独立 worktree。
2. **权限靠双层硬拦**。第一层 toolset 裁剪（角色根本没有越权工具），第二层 `pre_tool_call` hook 兜底（可真正 block 工具调用）。
3. **主干用 Kanban，不用 delegate_task**。跨角色、长周期、需审计、需恢复的工作全走 Kanban；delegate_task 只做单角色内部短推理。
4. **持久推进靠 Kanban dispatcher + 平台 watchdog**，不靠 `/goal` 做主循环；`/goal`/goal-mode 只在单个 worker 内部"跑到达标"。
5. **需求必须分两层**：core_need（MVP 必达）+ extended_need（可裁剪）；不确定时默认只交付 core_need。

## 2. 三层隔离模型（防污染的核心）

| 层级 | 机制 | 官方依据 |
|---|---|---|
| **进程级** | 每项目一个独立 Hermes 实例 + 独立 OS 用户 + 目录 `/data/projects/{id}` | Kanban 是单主机设计，kanban.db 是本地 SQLite，dispatcher 同机 spawn，不能跨实例共享 |
| **状态级** | 每项目独立 `HERMES_HOME`；config/记忆/session/skills/cron 全部按目录隔离 | profile = 独立 Hermes home 目录，119+ 文件经 get_hermes_home() 自动作用域到该目录 |
| **看板级** | 每项目一块 board；worker 由 `HERMES_KANBAN_BOARD` pin 住，只能看本 board | 官方："每 board 隔离是绝对的"，跨 board 链接被禁止 |
| **执行级** | Dev/QA 跑 Docker backend + 独立 worktree，只挂本项目目录 | 官方明示：profile 不沙箱，local backend 仍有用户全部文件访问权限 |
| **记忆级** | 每角色 profile 独立 MEMORY.md（2200 字符上限）；跨项目经验沉淀为 skill | 记忆是 profile 级单文件，多项目共用必爆且互相污染 |

## 3. 角色设计与权限（双层强制）

### 3.1 权限实现 = toolset 裁剪 + pre_tool_call hook

- **第一层**：`agent.disabled_toolsets` 显式移除危险内置工具集。**真机实测要点**：Hermes 的 `toolsets` 是"附加列表"，内置 `code_execution/terminal/file` 默认始终可用，必须用 `disabled_toolsets`（JSON 数组）显式禁用——CEO 禁 `code_execution/terminal/file`，设计角色禁 `code_execution/terminal`，物理上无法执行/写码。
- **第二层**：`policy-plugin` 注册 `pre_tool_call` hook，返回 `{"action":"block","message":...}` 兜底拦截（须 `hermes plugins enable policy` + `HERMES_ACCEPT_HOOKS=1` 才生效）。官方有现成例子：检查目标文件是否在 allowed_paths，不在就 block。

> **两个实现要点（早期版本踩过坑，务必注意）**：
> 1. **角色识别**。官方文档证实：每个 profile 有**独立的 HERMES_HOME**，形如 `…/.hermes/profiles/<name>`，运行该 profile 时 `HERMES_HOME` 被设到此子目录，故 `Path(HERMES_HOME)` 中 `profiles/` 后那一段即角色名（不存在 `HERMES_PROFILE` 环境变量，这是文档化的当前-profile 信号）。`resolve_role()` 优先用 hook kwargs/env，回退解析此路径段。**唯一需真机确认的**：本平台用「每项目自定义 HERMES_HOME 基目录」时，dispatcher spawn worker 是否仍把 HERMES_HOME 设到 `<base>/.hermes/profiles/<role>`（由《03》手册 Step 8 第 5 项验证）。即便此点不成立，插件 **fail-closed** 也只会过度拦截、不会放行。
> 2. **设计角色要能写 `design/`**。第二层不能把 `write_file` 对设计角色一刀切禁止，否则 synthesizer 写不了 `approved_versions.txt`（开闸钥匙），设计闸门死锁。正确边界：no-code 角色只硬拦 `terminal`/`patch`，`write_file` 限制到 `design/` 目录内。

### 3.2 角色权限表

| 角色 | profile 名 | 允许 toolset | 禁止 | 职责 |
|---|---|---|---|---|
| **CEO** | `ceo` | clarify, kanban, memory, messaging | **code_execution, terminal, file**（经 `disabled_toolsets` 禁用） | 只与用户沟通、拆派任务、检查回传证据、汇报；**绝不亲自干活** |
| **产品负责人** | `pm-lead` | kanban, file(只写 design/), memory | terminal, patch | 编排产品委员会，不亲自写 |
| **产品研究员 A/B** | `pm-research-a/b` | web, file(只写 design/) | terminal, patch | 不同模型查资料/竞品/反例 |
| **产品评审** | `pm-critic` | file(读全部), kanban | terminal, patch | 专门质疑需求遗漏与过度设计 |
| **架构负责人** | `arch-lead` | kanban, file(只写 design/) | terminal, patch | 编排架构委员会 |
| **架构设计师** | `arch-simple/scale/security` | file(只写 design/) | terminal, patch | 不同角度（轻量/扩展/安全）出方案 |
| **架构评审** | `arch-critic` | file(读全部) | terminal, patch | 找耦合、找"改一处影响多处" |
| **研发总监** | `dev-lead` | kanban, file(读全部) | patch 业务代码 | 切分任务、链接依赖，不亲自写 |
| **工程师** | `dev-worker-*` | terminal(Docker), file/patch(**仅 task worktree 内**), kanban | 改 worktree 外文件 | 实现分配的 task |
| **质控** | `qa` | terminal(Docker), file(写测试), kanban | 改业务代码 | 写/跑测试、阻断发布、生成缺陷任务 |
| **发布** | `release` | terminal, kanban | QA 未过时操作 | 仅 QA gate 通过后合并/打包/部署 |
| **变更管控** | `change-guardian` | file(读全部、写 design), kanban | 改代码 | 变更影响分析与设计闸门 |

每个 worker profile 用 `--description` 注册，供 Kanban 编排路由识别。

## 4. 需求双层结构（你的硬要求）

项目创建后，CEO 必须先推动产品委员会产出 `/workspace/design/requirements.yaml`：

```yaml
core_need: |
  用户最核心诉求，MVP 必须满足。这是项目成败的唯一判据。
extended_need: |
  大而全 / 增强功能 / 未来版本。明确标记"可裁剪"。
non_goals: |
  明确不做的部分。
acceptance_core:   # 核心验收标准，逐条可测试、可判定
  - ...
acceptance_extended:  # 扩展验收标准
  - ...
```

**与 goal 机制绑定**：
- 项目总 goal-mode 卡的验收标准 **只锁 `acceptance_core`**。
- `extended_need` 用 `/subgoal` 在 core 达成后追加，不重置循环；目标直到 core 与所有已追加 subgoal 都满足才算完成。
- **用户不确定要大而全还是最小化时，默认先交付 core_need，extended_need 进 backlog**。CEO 把分层结果展示给用户确认，不替用户猜。

## 5. 设计委员会：多模型头脑风暴 + 互相质疑（你的硬要求）

设计阶段不靠单个 agent。用 Kanban Swarm 拓扑 fan-out，**不同 profile 配不同大模型**，结果可追踪、可重试、可恢复。

### 5.1 产品设计委员会

```text
pm-lead（编排，不亲自写）
  ├─ pm-research-a（模型 X）：资料/竞品/用户场景
  ├─ pm-research-b（模型 Y）：专查反例和风险（换模型避免同质化）
  ├─ pm-critic（模型 Z）：专门质疑需求遗漏与过度设计
  └─ pm-synthesizer：汇总 PRD v1 → /workspace/design/PRD.md
```

### 5.2 架构设计委员会

```text
arch-lead（编排，不亲自写）
  ├─ arch-simple：最轻可落地方案
  ├─ arch-scale：扩展性方案
  ├─ arch-security：安全/隔离/权限审查
  ├─ arch-critic：找耦合、找"改一处影响多处"
  └─ arch-synthesizer：生成 ADR + 接口规范 + 代码规范 + TODO v1
```

整轮设为 **goal-mode 卡**：验收标准 = 通过 critic 无重大异议。没通过就继续迭代，把方案在开工前打磨到极致，最大限度减少后期返工。

> 实现要点：用 `hermes kanban swarm "<设计目标>" --worker pm-research-a:pm-research-a --worker pm-research-b:pm-research-b --verifier pm-critic --synthesizer pm-synthesizer`，共享上下文存在根卡的 JSON 评论（blackboard）里供所有 worker 读取。

## 6. 先设计再执行（你这次最强调的纪律）

固化为**平台状态机强制流程 + hook 硬拦**，不靠模型自觉。

### 6.1 变更处理流水线

```text
需求变更（用户提出 或 worker 遇阻）
  → 建 change-request 卡，assignee = change-guardian
  → change-guardian（goal-mode）影响分析：列出所有受影响文件/接口/模块
  → 先更新设计：PRD / ADR / API / 代码规范 / TODO
  → 重新生成任务图（Kanban 任务 DAG），标记旧任务 stale/cancelled
  → 新任务带新 design_version 执行
  → 全量 QA gate（受影响范围的完整回归，不是局部测试）
```

### 6.2 Dev worker 的 pre_tool_call 硬校验（三道闸）

每次 Dev 调 terminal/patch/write_file 前，hook 检查，任一不满足直接 block：
1. 当前 task 是否有 **approved design_version**（未批准的设计不准动代码）；
2. 是否绑定合法 **task id**；
3. 目标文件是否在该 task 的 **allowed_paths** 内（防止改一处波及别处，规范化后判断防 `..`/symlink 逃逸）。

### 6.3 QA gate 硬闸（QA 未过不得发布）

把"QA 阻断"从口头约定固化为代码强制：
- **QA** 完成后必须写 `reports/qa/status.json`（`release_allowed` / `failed` / `waivers[审批来源]`）；`failed>0` 且无 waiver 时 `release_allowed` 必须为 `false`。
- **release** 的任何 terminal/写文件，hook 先校验 `reports/qa/status.json` 的 `release_allowed==true`，否则直接 block（文件缺失/解析失败/未放行一律拦）。
- QA 只能写 `tests/`、release 只能写 `dist/`，都不得改业务代码。

> 这样平台不会把"部分失败但理由看起来合理"误判为可发布——失败必须显式 waiver 且注明审批来源。

## 7. "90 轮自动停"的处理（你特别关心的细节）

确认：`agent.max_turns` 默认 **90**，预算耗尽时先生成已完成工作总结再停；子 agent 预算独立（默认 50）。

四层应对：
1. **任务拆小**（根本解）：每张 Kanban 卡目标控制在 20–50 次工具调用内。
2. **调高预算**：worker profile 设 `agent.max_turns: 200`。
3. **goal-mode 卡片**：长任务卡用 `--goal`，在 Ralph 循环里跑，单卡不受单轮 90 硬限；预算耗尽会 block 卡片等人工，而非静默失败。
4. **平台 Watchdog**：监听 Kanban 事件流，遇 `gave_up`/`stale`/`timed_out`/`protocol_violation` 自动建 continuation 卡，不等用户输入。
5. **heartbeat 纪律**：worker 超 1 小时必须每小时 `kanban_heartbeat`，否则被当崩溃回收丢进度。

## 8. Web 与对外 API（只对接 CEO）

Web 与外部程序都只调你的网关，**绝不直连 Hermes 的 `/v1/*`**（直连等于暴露 terminal/file/web 全工具）。

```http
POST   /api/projects                      创建项目（建实例+board+profiles）
POST   /api/projects/{id}/messages        与 CEO 多轮沟通需求
GET    /api/projects/{id}/events          SSE 事件流（看板/日志）*
GET    /api/projects/{id}/requirements    读需求双层结构
POST   /api/projects/{id}/confirm-plan    用户确认方案 → 启动自动执行
POST   /api/projects/{id}/change-requests 提交变更（触发设计闸门）
GET    /api/projects/{id}/tasks           读 Kanban 看板
GET    /api/projects/{id}/artifacts       阶段成果产物
GET    /api/projects/{id}/demo            阶段性可运行成果
```

> \* 当前实现的 `/events` 为最小版：每次连接推送一次看板快照即结束。真正的持续 SSE 流属生产化项（阶段 13），由 Redis 事件总线订阅后持续推送。

**安全红线**：Hermes dashboard 的 `/api/plugins/*` 默认无鉴权且只绑 localhost，**绝不 `--host 0.0.0.0` 暴露**。对外鉴权全在你的 FastAPI 网关。

## 9. 关键风险与规避（落地前必读）

| 风险 | 后果 | 规避 |
|---|---|---|
| 子 agent / worker 不按指定模型运行（历史 bug） | 用错模型、上下文不足而失败 | 阶段 2 必须实测每个 profile 真实运行模型，不通过不上线 |
| scratch 工作区任务完成被删 | 代码产物丢失 | 一律用 `--workspace worktree:<path>` 或 `dir:<绝对路径>` |
| profile 误当沙箱 | worker 改到项目外文件 | Dev/QA 强制 Docker backend + 显式 terminal.cwd + hook 校验 allowed_paths |
| 多项目共用 ~/.hermes | 记忆爆/互相污染 | 每项目独立 HERMES_HOME |
| Kanban 跨主机 | 不支持 | 单机部署；要多机则每机独立实例 + 队列桥接 |
| dashboard 0.0.0.0 暴露 | 无鉴权工具被外部调用 | 只绑 localhost，对外走 FastAPI 网关 |
| worker 超 1 小时无 heartbeat | 被当崩溃回收丢进度 | worker 每小时 kanban_heartbeat |

## 10. 是否转 Claude Code 的决策框架

- **Hermes 独特价值**：原生多 profile + Kanban 持久看板 + 断点续跑 + 跨会话记忆/技能自进化 + 模型无关（可混用不同模型做质疑）+ 自带 dashboard 与 OpenAI 兼容接口 + 可硬拦工具的 hook。你的需求几乎全是原生能力，自研量小。
- **代价**：0.16 高速迭代（距上版 874 提交、399 issue 关闭），稳定性需自验；单主机；dashboard 默认无鉴权要补；子 agent 模型指定历史有坑。
- **Claude Code 权衡**：编码单点强、稳定，但缺多 profile/看板/持久队列/工具拦截 hook 这类编排原语，你那套多角色编排基本得从零自研。
- **建议**：先在 Hermes 上跑通操作手册第 12 步的"端到端验证六点"。全过则 Hermes 边际成本明显更低；若稳定性或子 agent 模型指定反复翻车，再评估"Claude Code 做编码执行 + 自研轻量编排"的混合路线。**决策依据是真实验证结果，不是文档承诺。**
