# 03 · 本地全流程部署与验证手册（step-by-step）

> 配套《01-最终设计方案.md》《02-从零开始操作手册.md》。本手册带你在**一台真实 Ubuntu 主机**上，
> 从零部署到让一个 demo 项目**完整跑通**，并验证设计既定目标。

> **能不能在云端/CI 里跑？不能。** 真实运行需要 `hermes` 程序 + Docker + 模型供应商凭证 + 出站网络，
> 四者缺一不可。请在你自己的主机上执行。
>
> **图例**：🧑‍💻 = 需要你亲自介入/决策　·　`命令`　·　✅ = 预期输出　·　☐ = 打勾位

> [`docs/hermes-验证矩阵.md`](docs/hermes-验证矩阵.md)）。仍标 ⚠️ 的少数 flag
> （如 `--goal-max-turns`、`--body`、`--parent`、部分配置键名）请先用 `hermes <cmd> --help`
> 确认；不符时按 `--help` 调整。`kanban swarm`、`boards create --switch`、

## A. 开始前你必须准备/配置的东西（🧑‍💻 全部需要你）

| # | 必备项 | 说明 |
|---|---|---|
| A1 | **Ubuntu 22.04+ 主机** | ≥8 核 / 32GB 内存 / 200GB 磁盘，有 sudo，能联网 |
| A2 | **模型供应商凭证（最关键）** | `hermes setup --portal`（Nous Portal OAuth，一次配齐）或 `hermes setup` 填 OpenRouter/Anthropic/OpenAI key。**没有它，一切都跑不起来。** 多模型质疑建议用 OpenRouter（一个 key 覆盖多模型） |
| A3 | **确认模型/供应商** | 两个大模型：**GLM-5.2（默认）+ DeepSeek**（`glm-5.2`/`deepseek-chat`），用环境变量 `ZAI_PRIMARY_MODEL`/`DEEPSEEK_MODEL`/`*_BASE_URL` 覆盖；key 放 `~/.hermes/.env`。`hermes model` 看可选（见 Step 2.5） |
| A4 | **一个告警通道** | `ALERT_WEBHOOK_URL`（Slack/飞书/钉钉自定义机器人）或 `ALERT_EMAIL`（主机需装 `mail`）。监测脚本用 |
| A5 | **PLATFORM_TOKEN** | `openssl rand -hex 16`，控制平面对外鉴权用，自己保管 |
| A6 | **想清楚 demo 的 core_need** | Step 5 你要扮演用户跟 CEO 说需求。准备一句话核心诉求，例如"命令行待办工具，增删查改，存本地文件" |

## Step 2 · 部署平台文件 + 沙箱镜像 + 控制平面服务
**目的**：铺文件到 `~/platform`、`~/platform-base`，建 venv，构建非 root 沙箱镜像，并把**控制平面装成 systemd 服务**启动。

```bash
# 默认 production 模式（保留人工 review）；无人值守用：
AUTOCODE_MODE=unattended ./scripts/01-deploy-platform.sh
# 或显式生产：
./scripts/01-deploy-platform.sh
```
✅ 预期：`~/platform/`（含 `*.py`、`*.sh`、`venv`、`.platform_token`、`.platform_runtime.env`）、`~/platform-base/templates/`（17 个 SOUL）均存在；`docker images | grep autocode-python` 有沙箱镜像；构建后自检打印"✅ 沙箱镜像就绪，git CLI 可用"；末尾打印"控制平面已作为 systemd 服务运行在 127.0.0.1:9000"与运行时模式。

☐ Step 2 通过

> 🔒 **安全/无人值守说明**：
> - **沙箱镜像是安全前提**：dev-worker 跑在非 root 自定义镜像里（产物属主正确、爆炸半径受控），且**镜像内装了 git**——worktree 并行开发/分支合并依赖它。若镜像缺失或无 git，部署会**拒绝继续**（不回退公共 root 镜像）；仅本地调试可 `ALLOW_PUBLIC_SANDBOX_FALLBACK=1`。
> - **运行时模式 `AUTOCODE_MODE`**：`production`（默认，保留人工 review）/ `unattended`（自动放行 review，安全靠 QA gate + 设计闸门 + 交付完整性闸门三重兜底）/ `demo`（同 unattended 但标记不可生产）。写入 `~/platform/.platform_runtime.env`，三个自动化定时器共读。
> - **无人值守审批**：`AUTOCODE_UNATTENDED=1`（默认）让本项目跳过 DANGEROUS/Tirith 审批（否则自动化被 approval 反复打断）。**安全不降级**——HARDLINE 12 条 + sudo-stdin guard 仍不可绕过，CEO 无终端、worker 在容器；配置只写本项目 HERMES_HOME。需保留人工审批则 `AUTOCODE_UNATTENDED=0`。详见验证矩阵 §G。
> - **Docker 权限**：当前用户须能 `docker info`（在 docker 组）。部署会早失败并提示 `newgrp docker` 或重新登录；**不要 `chmod 666 /var/run/docker.sock`**（把 Docker root 权限暴露给全机用户）。

## Step 6 · 确认方案（落盘需求）→ 启动产品委员会
**目的**：把你与 CEO 对齐的需求双层结构**落盘**为 `requirements.yaml`，再显式建产品委员会 swarm。
> ⚠️ 关键链路：CEO 没有 file 工具集，**写不了 `requirements.yaml`**；而产品委员会以它为输入。所以必须由网关在 confirm-plan 时落盘——把结构放进请求体传入。

```bash
curl -s -X POST http://127.0.0.1:9000/api/projects/demo1/confirm-plan \
  -H "X-Token: $TOKEN" -H "Content-Type: application/json" \
  -d '{"requirements":"core_need: 命令行待办工具，能增删查改，数据存本地文件\nextended_need: 优先级排序/多端同步（可裁剪）\nnon_goals: 不做图形界面\nacceptance_core:\n  - 能 add/list/done/rm\n  - 进程重启后数据仍在\n"}' | jq
# 确认已落盘
curl -s http://127.0.0.1:9000/api/projects/demo1/requirements -H "X-Token: $TOKEN" | jq
export HERMES_HOME=/data/projects/demo1/.hermes
hermes kanban --board demo1 list
```
✅ 预期：confirm-plan 返回 `{"status":"plan-confirmed","swarm":"product-council",...}`；`/requirements` 回显你写入的 YAML；看板出现 **worker 卡（pm-research-a/b）+ verifier 卡（pm-critic，gated）+ synthesizer 卡（pm-synthesizer，gated）**，dispatcher 每 60s tick 推进。

☐ Step 6 通过（requirements 已落盘 + swarm 已 fan-out）

## Step 7 · 设计开闸 + 构建流水线
**目的**：产品委员会出 PRD → 架构委员会出 ADR → 开"设计闸门钥匙" → 研发并行实现。
> 这一整段（产品→架构→dev→QA→release）由 **`orchestrator.py` 状态机自动编排**（Step 9 的 cron 每分钟 tick，幂等推进）。也可用端点 `POST /architecture-swarm` 或下面的手动命令单独起架构 swarm。

```bash
export HERMES_HOME=/data/projects/demo1/.hermes
WS=/data/projects/demo1/workspace

# 7.1 等产品委员会跑完（数分钟，dispatcher 60s tick），确认 PRD 产出
ls "$WS/design/"                                  # ✅ 预期出现 PRD.md

# 7.2 启动架构委员会 swarm —— 推荐用端点（或等 watchdog 自动起）：
curl -s -X POST http://127.0.0.1:9000/api/projects/demo1/architecture-swarm -H "X-Token: $TOKEN" | jq
# 或手动等价命令：
hermes kanban --board demo1 swarm \
  "产出 ADR+接口规范+代码规范+TODO：基于 design/PRD.md" \
  --worker arch-simple:arch-simple --worker arch-scale:arch-scale --worker arch-security:arch-security \
  --verifier arch-critic --synthesizer arch-synthesizer

# 7.3 等架构通过，确认"设计闸门钥匙"已写（若 arch-synthesizer 未自动写，人工补）
cat "$WS/design/approved_versions.txt" 2>/dev/null || echo "v1" > "$WS/design/approved_versions.txt"
ls "$WS/design/" | grep -E 'ADR|allowed_paths'    # ✅ 预期 ADR.md + allowed_paths.<task>.txt

# 7.4 dev-lead 切任务（或手动建编码卡），dispatcher 自动 spawn dev-worker 实现、qa 跑测试
watch -n 30 "HERMES_HOME=$HERMES_HOME hermes kanban --board demo1 list"
```
✅ 预期：编码卡 ready→in_progress→done，`$WS/src/` 出现代码产物，qa 卡在全部编码卡 done 后跑。

> **QA gate**：qa 完成后写 `$WS/reports/qa/status.json`（`release_allowed` 等）；release 受 policy 插件硬拦——无 `release_allowed:true` 时 release 的 terminal/写文件全被 block（验证：`cat $WS/reports/qa/status.json`；失败测试无 waiver 时 `release_allowed` 应为 false）。

☐ Step 7 通过（PRD/ADR 产出、闸门开、产物落地、QA gate 把关）

## Step 8 · 端到端验证（阶段 12 六项，**核心**）
逐项跑，全过才算落地。前两项+第 5 项同时验证了三个"真机不确定点"。

> **一键复验**：`scripts/verify-acceptance.sh demo1 demo2` 会自动判定隔离/CEO 不干活/并行不冲突/设计闸门四项并输出 JSON 汇总；自动推进与跨 90 轮需观察长跑（脚本标 `manual`）。下表是逐项手动口径与命令。

| # | 目的（验证点） | 命令 | ✅ 预期 | ☐ |
|---|---|---|---|---|
| 1 | **隔离**（gateway/board 隔离） | 再建 `demo2`；`ls /data/projects/demo{1,2}/.hermes/kanban*`；`hermes kanban --board demo1 list` 与 `--board demo2 list` 对比 | 两套独立 `kanban.db`；两个 `autocode-gw-*.service`；互相看不到对方任务 | ☐ |
| 2 | **CEO 不干活**（第一层权限+角色解析） | `grep -A3 disabled_toolsets /data/projects/demo1/.hermes/profiles/ceo/config.yaml`（注：`config show` 不渲染该字段，读 config.yaml）；再让 CEO"自己写段代码" | 输出含 `code_execution,terminal,file`；CEO 拒绝并改为建卡派活 | ☐ |
| 3 | **自动推进** | 确认方案后不再人工干预，观察看板 | goal-mode + dispatcher 跑到 core_need 达成 | ☐ |
| 4 | **并行不冲突** | 看 dev-worker-1/2 各自 worktree：`git -C $WS worktree list`、`ls $WS/.worktrees`、`git -C $WS rev-list --all --count` | 各任务独立 worktree、各自分支提交，无文件互相覆盖（提交数 >1） | ☐ |
| 5 | **设计闸门**（hook 触发 + 角色识别 + **task_id 可达**，#25204） | 见下方确定性脚本 (a)+(b) | (a) 无 approved 时被拦；(b) 补 approved+allowed_paths 后放行 | ☐ |
| 6 | **跨 90 轮** | 给一个需 >90 工具调用的任务 | goal-mode/watchdog 平滑续跑而非静默停 | ☐ |

> **第 4 项的前置**：沙箱镜像必须含 git（Step 2 已自检）+ workspace 已持久化 git 身份；dev-lead 为每张编码卡指定 `--workspace worktree:${WORKTREE_ROOT}/<短名>`，dev-worker 完工 `git commit`。否则 worktrees 为空、产物互相覆盖（第四轮真机 P0）。另：**交付完整性闸门**会在 release 前独立校验"dev 卡 done 是否真有提交/源码落地"，挡 agent 谎报。

**第 5 项确定性验证（务必做，一次验证三个命门：hook 触发 / 角色识别 / task_id 可达）**：
```bash
export HERMES_HOME=/data/projects/demo1/.hermes
WS=/data/projects/demo1/workspace

# (a) 拦截半场：故意制造"无 approved design"，预期写码被 block
mv "$WS/design/approved_versions.txt" /tmp/av.bak 2>/dev/null || true
hermes kanban --board demo1 create "GATE TEST A: write src/gate_probe.py" \
  --assignee dev-worker-1 --idempotency-key gate-test-a \
  --body "Use write_file to create src/gate_probe.py" --goal
sleep 200
test -f "$WS/src/gate_probe.py" && echo "❌ (a) 闸门失效：文件被写出！" || echo "✅ (a) 闸门生效：未写出"

# (b) 放行半场：补回 approved + 为这张卡写 allowed_paths，预期放行
#     —— 这一步专门验证【task_id 是否真能到达 hook】(问题A)。
mv /tmp/av.bak "$WS/design/approved_versions.txt" 2>/dev/null || echo "v1" > "$WS/design/approved_versions.txt"
TID=$(hermes kanban --board demo1 create "GATE TEST B: write src/gate_probe_b.py" \
  --assignee dev-worker-1 --idempotency-key gate-test-b \
  --body "Use write_file to create src/gate_probe_b.py" --goal --json | jq -r '.id')
echo "src/" > "$WS/design/allowed_paths.${TID}.txt"     # 声明本卡可写范围
sleep 200
test -f "$WS/src/gate_probe_b.py" \
  && echo "✅ (b) 放行成功：approved+allowed_paths 下写入成功 → task_id 已到达 hook" \
  || echo "❌ (b) 仍被拦：approved+allowed_paths 齐备却写不出 → hook 拿不到 task_id（问题A 成立）"
```
✅ 预期：(a) 打印 **✅ 闸门生效**，(b) 打印 **✅ 放行成功**。
- (a) ❌ → hook 在 kanban-worker 路径未触发（issue #25204），第二层失效，靠第一层 `disabled_toolsets` 兜底，上线前必须解决。
- (b) ❌（而 a 正常）→ **hook 拿不到 task_id**（验证矩阵 B 节第二大不确定点），第三道闸"全锁"，需改用 Hermes 原生审批或换 task_id 注入方式。

> **第 5 项一次性验证三个真机命门**：hook 是否触发（a）、自定义 HERMES_HOME 下角色解析（a/b 任一通过即成立）、**task_id 是否可达**（b）。

☐ Step 8 六项全过

## Step 9 · 部署监测与告警（盯住"改不了、只能监测"的部分）
**目的**：对 hook 失效、gateway 宕、任务卡死、权限漂移等做持续监测 + 主动告警。

**自动化三循环已由部署脚本装好。** `scripts/01-deploy-platform.sh` 会把
orchestrator/watchdog/monitor 装成 `systemd --user` 定时器（与控制平面同机制：
开机自启、失败有 journald 日志），无需手动配 cron。确认：
```bash
systemctl --user list-timers 'autocode-*'
# 预期看到 autocode-orchestrator.timer（每分钟）/ autocode-watchdog.timer（每分钟）
#         / autocode-monitor.timer（每 5 分钟）三个 ACTIVE
journalctl --user -u autocode-orchestrator.service -n 20 --no-pager   # 看状态机近期 tick
```
监测告警地址：编辑 `~/platform/.platform_token.env` 追加 `ALERT_WEBHOOK_URL=...`
（三个定时器共用该 EnvironmentFile），再 `systemctl --user restart autocode-monitor.timer`。

**hook_canary 需手动加**（重型探针，不在自动三循环内）：
```bash
# hook_canary：每小时一次，自动探测设计闸门是否仍生效
( crontab -l 2>/dev/null; echo "0 * * * * ALERT_WEBHOOK_URL='https://你的webhook' ~/platform/hook_canary.sh demo1 >> ~/platform/monitor.log 2>&1" ) | crontab -
```

> 若所在主机无 systemd（容器等），可改用 cron 等价物：

## 附 2 · 监测告警覆盖表（哪条监测盯哪个风险）
| 风险（多为"改不了、只能监测"的 Hermes 行为） | 监测手段 | 触发动作 |
|---|---|---|
| 任务卡死/超时/gave_up/stale/protocol_violation | `watchdog.sh`（每分钟） | 自动建续跑卡（带 `--idempotency-key` 去重、继承 assignee/workspace） |
| gateway 宕 | `monitor.sh` check_gateway | CRIT 告警 |
| 卡死任务持续堆积（watchdog 兜不住） | `monitor.sh` check_stuck | WARN 告警 |
| 第一层权限被改动（CEO 重获 terminal） | `monitor.sh` check_first_layer | CRIT 告警 |
| Hermes 崩溃（日志 Traceback/CRITICAL） | `monitor.sh` check_logs | WARN 告警 |
| 数据盘将满 | `monitor.sh` check_disk | WARN 告警 |
| Docker root 写入致宿主读不了产物 | `monitor.sh` check_root_files | chown 归还属主 + WARN |
| 策略闸门降级（拿不到 task_id 走 taskless 兜底，细粒度隔离退化） | `monitor.sh` check_policy_fallback | WARN 告警（读 `reports/security/policy_fallback.jsonl`） |
| 供应商限流（1305）/余额耗尽（1113） | `monitor.sh` check_logs | 写 `.provider_pause` 暂停续跑 + WARN/CRIT |
| **设计闸门 hook 失效（#25204）** | `hook_canary.sh`（每小时） | CRIT 告警（单边探针：抓确定失效） |

> 监测脚本本身的局限已在脚本头注明：`hook_canary.sh` 是单边探针，能抓"确定失效"但不能证明"一定正确"，首次上线仍以 Step 8 第 5 项人工确定性验证为准。
