# 第七轮整体优化方案（基于 shi 真机失败 + 测试组交付包 + GPT 评审）

> 输入：测试组交付包 `autocode-delivery.tar.gz`（含改过的 launch/control_plane/orchestrator/webui/SOUL + shi 测试报告 + 问题清单）+ GPT 评审建议。
> 基线：当前 `main`（`e70be29`）。原则：**采纳问题判断，按生产级重写修复，不整包覆盖、不头疼医头**。

## 0. shi 真机暴露的硬事实（不是体验瑕疵，全部命中项目目标）
13 小时跑 `shi`（古诗搜索）项目：**9 次人工干预**、stage 虽 complete 但代码是 CEO 越权写的、dev-worker 全因容器挂载错误失败被强制 done、QA 没写 status.json、release 超时被强制完成。逐条对应项目核心目标的失守：

| 失守点 | 对应目标 | 本轮修法（生产级，非补丁） |
|---|---|---|
| Docker 容器把 demo2 的 workspace 挂进 shi | 多项目隔离 | A3 注入 `AUTOCODE_PROJECT_ID` + monitor `check_docker_mount_isolation`（容器声明项目≠实际挂载→CRIT）+ 验收点 2b |
| CEO 在 confirm-plan 前用 `execute_code` 直接写业务代码 | CEO 只沟通不编程 | A1 两层封堵 `execute_code`（disabled_toolsets + policy CODE/SENSITIVE_TOOLS）+ monitor + 验收 |
| orchestrator 重启后 timer 失效，全靠人工 tick | 无人值守自动推进 | B1 把 orchestrator 内嵌进**控制平面常驻进程**（lifespan + 跨进程锁，与 timer 互斥） |
| arch-synthesizer 没写 approved_versions.txt | 设计闸门/自动推进 | B2 **不放宽**闸门，改建"补齐批准文件"卡 + 强化 SOUL |
| QA 没写 reports/qa/status.json | QA 阻断/自动推进 | B3 建"补齐 QA 结论"卡（assignee=qa）+ SOUL 收尾自检 |
| release 超时（改业务代码触发 scope_guard） | 职责边界/自然完成 | C1 SOUL.release 收窄 + 必出 manifest；complete 以 manifest 为准 |
| UI 无"确认需求"按钮，用户手写 requirements.yaml | 用户经网关对接 CEO | D1 webui 加"✅ 确认需求并启动设计"按钮 + D2 confirm-plan 幂等+readback |

## 1. 对交付包的取舍（与 GPT 评审一致，逐项判定）
- **直接采纳（判断+方向都对）**：禁 `execute_code`；YOLO 默认 0；QA status 缺失检测；release 职责收窄+manifest；confirm-plan 系统反馈；UI 确认需求按钮；`/deliverable` 端点方向。
- **重写后采纳（思路对、实现粗糙）**：
  - Docker 清理 → **不**整批删 `hermes-*` 容器（会杀别的项目，且交付包的"必须挂当前项目"判据对多项目是误杀）；改为**声明项目 vs 实际挂载**一致性检测。
  - 内嵌 orchestrator → 加**开关（默认部署开）+ 跨进程锁 + to_thread + lifespan 优雅起停**，不裸跑 async task、不与 timer 双跑。
  - PRD/ADR 文件名兼容 → 采纳但**标降级**：canonical 优先，兜底匹配 `design/*prd*.md` 并落 `.autocode/warnings.jsonl`。
- **不采纳（会削弱设计/安全）**：全局删所有容器；`approved_versions` 宽松匹配 `*approved*`（`not-approved-yet.md` 会误开闸）；blocked 超时无条件 unblock（绕过 critic 真实阻断）；CSP 加 `unsafe-eval`。
- **本轮暂缓（独立任务再做）**：文件上传端点（+python-multipart，需限大小/类型/白名单）；Vue/Element Plus UI 重写（缺 static 资源、需浏览器验证）；blocked 细粒度分类（当前已无危险的无条件 unblock，够用）。

## 2. 落地清单（已实现并测试）
**安全/隔离（P0）**：`launch_project.sh`（CEO/no-code/executor 禁 execute_code、注入 PROJECT_ID、YOLO=0）、`policy_plugin.py`（CODE/SENSITIVE_TOOLS 含 execute_code）、`monitor.sh`（check_first_layer 双查、check_docker_mount_isolation）、`verify-acceptance.sh`（验收点 2 双查 + 2b Docker 隔离）。
**可靠自愈流水线（P1）**：`orchestrator.py`（tick_lock 跨进程锁、approved/QA-status repair 卡、PRD/ADR 容忍+警告、manifest 收紧 complete + completion_mode=natural）、`control_plane.py`（lifespan 内嵌 orchestrator，gated by AUTOCODE_EMBEDDED_ORCHESTRATOR，部署默认开）、`01-deploy`（控制平面 unit 写 AUTOCODE_EMBEDDED_ORCHESTRATOR=1）。
**交付收口/UI**：`SOUL.arch-synthesizer/qa/release`（必出 canonical 文件 + 自检 + manifest + 职责收窄）、`control_plane.py`（confirm-plan 幂等+readback+系统消息、`/deliverable` 端点）、`webui.html`（确认需求按钮）。

## 3. 验收（本轮）
- `make ci`：**171 测试全过** + shellcheck + 文档块校验；实操 e2e 谐振 **35/35**（含 CEO execute_code 被拦、manifest 收紧 complete）。
- 七项硬验收对齐 GPT 终版：①多项目隔离 ②Docker mount 隔离 ③CEO 禁 execute_code ④YOLO=0+hook ⑤confirm 后自动推进 ⑥QA 缺 status 自愈 ⑦complete=自然（manifest 存在 + completion_mode=natural）。

> 一句话：这一轮把 shi 真机暴露的两个 P0（跨项目挂载、CEO 越权 execute_code）和"无人值守不可靠"（timer 失效、缺 repair、release 不收口）按**生产级**系统性修掉——隔离与权限做成双层+监测+验收，自动推进做成"控制平面常驻进程内嵌 + 跨进程锁 + 自愈 repair 卡 + manifest 收口"，UI 补上 CEO→流水线的"确认需求"这一桥。交付包里粗糙/削弱安全的做法一律重写或不采纳。
