## 1. P0：建项目失败要回滚端口、回明确 502（D18 / D16）

### 问题（源码证据）
`platform/control_plane.py create_project`：
```python
port = ports.allocate(body.project_id)
project = gateway.launch(body.project_id, port)   # launch_project.sh 非 0 → CalledProcessError
registry.add(project)
```
`gateway.launch` 内 `subprocess.run(..., check=True)`。当 launch 失败（磁盘不足/缺 key/docker 不可用），异常未捕获 → FastAPI 返回**泛化 500**；更糟的是 `ports.allocate` 已把端口写进 `.ports.json`，**端口永久泄漏**成孤儿占用（D16：报告里 autosousuo:8651 / sou:8652 残留）。

### 改法
1. `PortAllocator` 增 `release(pid)`（文件锁内删 `assigned[pid]`，不回退 `next` 以免与并发分配撞端口）。
2. `create_project` 包 try/except：launch 失败 → `ports.release(pid)` 回滚 → 抛 `HTTPException(502, 具体 stderr)`。

```python
port = ports.allocate(body.project_id)
try:
    project = gateway.launch(body.project_id, port)
except Exception as exc:
    ports.release(body.project_id)
    detail = exc.stderr.strip() if hasattr(exc, "stderr") and exc.stderr else str(exc)
    raise HTTPException(502, f"failed to create project '{body.project_id}': {detail}")
registry.add(project)
```

### 测试 / 契约
- `tests/test_control_plane.py::test_create_project_rolls_back_on_launch_failure`：launch 抛错 → 502 + `.ports.json` 不残留该 pid。
- 静态契约：`control_plane.py` 含 `def release(` 与 `ports.release(`。

## 2. P0：kanban 子进程加超时（D20）

### 问题
`HermesGateway.kanban` 的 `subprocess.run` 无 `timeout`。`/api/projects`（Web UI 每 30s 刷新）会**逐项目**调 kanban；任一 hermes 进程卡住 → 整个列表请求挂起，FastAPI 同步线程池被占满。

### 改法
```python
try:
    out = subprocess.run(cmd, env=env, capture_output=True, text=True,
                         timeout=int(os.environ.get("KANBAN_TIMEOUT", "15")))
except subprocess.TimeoutExpired as exc:
    raise RuntimeError(f"hermes kanban timed out after {exc.timeout}s") from exc
```
超时转成现有的 `RuntimeError` 路径——`list_projects` 的 `_kanban_summary` 已 catch `RuntimeError` 返回空摘要，所以**单项目超时只让该项目摘要为空，其余正常加载**（不再整体阻塞/500）。

### 测试 / 契约
- `test_list_projects_survives_kanban_error`：kanban 抛 RuntimeError → `/api/projects` 仍 200，该项目 `task_summary={}`。
- 静态契约：含 `TimeoutExpired` 与 `KANBAN_TIMEOUT`。

## 3. 文档项（D11 / D6 / D7）

- **D11（`00-host-setup.sh`）**：`usermod -aG docker` 后提示三选一让 **user systemd** 拿到新组（重新登录 / `newgrp docker` / `sudo systemctl restart user@$(id -u).service`）——否则 gateway（user service）访问不了 docker。
- **D6/D7（`03` A7）**：新增"仅非交互/AI-agent 部署需免密 sudo"前置项：在**自己的真实终端**配 `sudoers.d/$USER` NOPASSWD；明确**不要**让 agent 自己 `echo 密码 | sudo -S`（Hermes 防注入会拦，且无 PTY 读不到密码）。

## 4. 判定为不采纳 / 不适用（附理由）

- **D19（`--goal` 缺值）**：当前仓库 `--goal` 在 orchestrator（dev-lead/qa/release 卡）与 `change_request` 一致地当**布尔 flag** 用，真机 auto-progress（swarm/卡推进）已跑通；报告的报错是其私有版本现象或误诊。**不盲目加值**，以免破坏已验证可用的路径。若后续 `hermes kanban create --help` 确认 `--goal` 需取值，再统一调整（含 orchestrator）。
- **D21/D22/D23（前端）**：针对用户**私有增强版 `webui.html`**（confirm 拦截、Promise.allSettled、CEO 写根目录交付）。本仓库的 `webui.html` 是另写的**最小安全版**：渲染全程 `textContent` 防 XSS、按 Tab 顺序 await（无 Promise.all 卡死问题）、且 **CEO 无 file/terminal 工具**（`disabled_toolsets`）按设计根本不能在聊天里写文件——D23 的"CEO 绕流水线交付"在本平台架构里不成立。故不并入这些前端补丁。
- **D17（ComfyUI JSON）**：来自 Hermes 的 skills 复制；本仓库 `platform-base/skills/` 为空，`launch_project` 的 `cp -rn skills` 在空目录下无副本，非本仓库缺陷。若将来放大体积 skills，建议改 symlink 或按角色裁剪，避免 ×17 复制。
- **默认模型切 DeepSeek（D12 建议2）**：用户已定"默认 GLM-5.2"；第五轮限流自愈修复后，GLM 限流可自动恢复，**不必切默认**。
