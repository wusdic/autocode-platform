## 2. 必改（P0）—— 改哪里、怎么改

### F1：沙箱镜像安装 git（+ 证书 + safe.directory）

**文件**：`docker/python-sandbox.Dockerfile`
**问题**：worker 在容器内 `git worktree/add/commit/merge` 时无 git CLI → 分支化并行从物理上不可能。
**改法**：在 `FROM`/`ARG` 之后、切换到非 root 用户 **之前**（apt 需 root）插入安装层；并为非 root 用户设 `safe.directory`（容器内看到宿主属主文件时 git 不会因 "dubious ownership" 拒绝操作）。

```dockerfile
FROM python:3.11-slim

ARG UID=1000
ARG GID=1000

# dev/qa worker 需要 git 做 worktree 分支/提交/合并（否则并行开发与"合并分支"形同虚设）。
# 仅装最小集：git + 证书；装完清缓存控制镜像体积。
RUN apt-get update \
 && apt-get install -y --no-install-recommends git ca-certificates \
 && rm -rf /var/lib/apt/lists/*

RUN groupadd -g "${GID}" autocode 2>/dev/null || true \
 && useradd -m -u "${UID}" -g "${GID}" autocode 2>/dev/null || true

# 容器内 UID 映射到宿主 UID，仍可能因 worktree 元数据属主触发 git "dubious ownership"，
# 显式信任工作区，避免 worker 的 git 命令被拒。
RUN git config --system --add safe.directory '*'

USER autocode
WORKDIR /workspace
```

> 注：不引入 build-essential 等重依赖，保持镜像精简；如将来需要编译扩展再按需加。

### F2：持久化 workspace 的 git 提交身份

**文件**：`platform/launch_project.sh`（Bug-3 初始化块，line 45–51）
**问题**：身份用 `git -c user.email=… commit` 临时传入，**未写进 repo config**。worker 在容器内 worktree 里 `git commit` 时会报 *"Please tell me who you are"* 而失败——即使装了 git（F1）也提交不了。
**改法**：把身份**持久化到仓库本地 config**，所有 worktree 自动继承。

```bash
# Bug-3：dev-worker 用 git worktree 工作区，workspace 必须是 git 仓库，否则 worktree 失败。
if [ ! -d "${WORKSPACE}/.git" ]; then
  git -C "${WORKSPACE}" init -q 2>/dev/null || true
  # 关键：身份要写进 repo config（不能只用 -c 临时传），否则 worker 在容器内
  # worktree 里 git commit 会因"无身份"失败 → DEV-2~N 全部无 commit、无法合并。
  git -C "${WORKSPACE}" config user.email "autocode@local" 2>/dev/null || true
  git -C "${WORKSPACE}" config user.name  "autocode" 2>/dev/null || true
  git -C "${WORKSPACE}" add -A 2>/dev/null || true
  git -C "${WORKSPACE}" commit -qm "init workspace" --allow-empty 2>/dev/null || true
fi
```

### F5：可观测——dev 卡 done 但无产物提交时告警

**文件**：`platform/monitor.sh`（新增 `check_dev_commits`，并在 `main` 循环里调用）。
**动机**：报告里"DEV-4 安全代码没落地、git 只有 2 条提交"全程无人发现。加一条监测，把"声称完成但 git 无对应提交 / `.worktrees/` 为空"暴露成 WARN。

```bash
check_dev_commits() {  # ⑧ dev 卡声称完成但无 git 产物：worktree 空或提交数异常少 → 告警
  local pid="$1" ws="${2}workspace"
  [ -d "${ws}/.git" ] || return 0
  local done_dev commits
  done_dev=$(HERMES_HOME="${2}.hermes" hermes kanban --board "${pid}" list --json 2>/dev/null \
    | jq '[.[] | select((.assignee // "")|startswith("dev-worker")) | select(.status=="done")] | length' 2>/dev/null)
  commits=$(git -C "${ws}" rev-list --all --count 2>/dev/null || echo 0)
  # 有 N(>=2) 个 dev 卡完成，却几乎没有新提交（仅 init），高度疑似"产物未落地/被覆盖"。
  if [ "${done_dev:-0}" -ge 2 ] && [ "${commits:-0}" -le 2 ]; then
    notify WARN "project ${pid}: ${done_dev} 个 dev 卡 done，但 git 仅 ${commits} 次提交——疑似产物未提交/worktree 未生效（查 ${ws}/.worktrees/）"
  fi
}
```

在 `main()` 的 per-project 循环里加：`check_dev_commits "${pid}" "${proj_dir}"`。

### F6（可选细化）：worktree 内的任务级 scope 文件

第二轮加的"从 worktree cwd 反解 task_id"只在 worktree 目录名恰为 `t_<id>` 时生效；而 §2-F3 约定 worktree 用**语义短名**（storage/cli），反解会失败 → 退回项目级兜底。这不破坏安全（worktree 物理隔离 + Docker + 项目级路径兜底仍在），但细粒度 allowed_paths 不生效。

**若要恢复细粒度**（可选）：让 `policy_plugin.enforce` 在 dev-worker 分支、cwd 为 worktree 时，额外接受 worktree 根目录下的 `.allowed_paths.txt` 作为范围源（由 dev-lead/dev-worker 写入）。这样范围与 worktree 解耦于 task_id。**优先级低**，可在 F1–F5 验证通过后再评估是否需要。
