## 1. P0：monitor.sh 限流"无限暂停"循环（D12，当前 main 仍存在）

### 问题（源码证据）

`platform/monitor.sh:74`：
```bash
jlog=$(journalctl --user -u "autocode-gw-${pid}.service" -n 500 --no-pager 2>/dev/null || echo "")
```
`-n 500` 取**最近 500 行**，与时间无关。后果链：
```
某次 429/1305 写进 journal
  → 每 5 分钟 monitor 都从最近 500 行里再次匹配到这条【旧】消息
  → 每次都重写 .provider_pause = now + 600s
  → 暂停永远刷新，orchestrator/watchdog 永远不起新任务
  → 用户被迫手动 rm .provider_pause（报告里 rm 了 10+ 次）
```

### 改法（`platform/monitor.sh` check_logs，§80 附近）

**(A) 只看最近 10 分钟日志**（与 5 分钟监测周期对齐，旧消息自然滑出窗口）：
```bash
# 旧：-n 500（扫历史，旧 429 反复触发新暂停 → 无限暂停循环，D12）
# 新：--since "10 min ago"（只看近窗，旧消息滑出，限流恢复后不再被旧日志拖住）
jlog=$(journalctl --user -u "autocode-gw-${pid}.service" --since "10 min ago" --no-pager 2>/dev/null || echo "")
```

**(B) 只在未处于活跃暂停时才写 pause**（避免近窗内旧消息把活跃暂停一直往后顺延）：
```bash
if grep -Eq "1305|temporarily overloaded" <<<"${jlog}"; then
  pf="${PLATFORM_DATA_ROOT}/.provider_pause"
  now=$(date +%s); until_e=0
  [ -f "$pf" ] && until_e=$(tr -dc '0-9' < "$pf" 2>/dev/null); until_e="${until_e:-0}"
  if [ "$now" -ge "$until_e" ]; then   # 仅当无暂停或已过期才新写，避免无限顺延
    echo "$(( now + ${PROVIDER_PAUSE_SECONDS:-600} ))" > "$pf" 2>/dev/null || true
    notify WARN "project ${pid}: 供应商临时过载，暂停起新 swarm ${PROVIDER_PAUSE_SECONDS:-600}s"
  fi
fi
```

**(C) 清理已过期的 .provider_pause**（housekeeping：orchestrator 已按 epoch 判定，功能上无害，但清掉残留文件让快照/语义干净，也免得 glm 脚本那种"手动清"）。在 `main()` 的 `check_disk` 前加：
```bash
_clear_expired_pause() {
  local pf="${PLATFORM_DATA_ROOT}/.provider_pause" now until_e
  [ -f "$pf" ] || return 0
  now=$(date +%s); until_e=$(tr -dc '0-9' < "$pf" 2>/dev/null); until_e="${until_e:-0}"
  [ "$now" -ge "$until_e" ] && rm -f "$pf" && echo "$(date -Is) [info] 清理已过期 .provider_pause"
}
```
并在 `main()` 里 `check_disk` 之前调用 `_clear_expired_pause`。

@app.get("/api/projects/{pid}/artifact-content")
def artifact_content(pid: str, path: str = Query(...), x_token: str = Header(None)):
    auth(x_token)
    project = get_project(pid)
    ws = Path(project.workspace).resolve()
    target = (ws / path).resolve()
    if not target.is_relative_to(ws):          # 防穿越（比 startswith 严谨）
        raise HTTPException(403, "path traversal denied")
    if not target.is_file():
        raise HTTPException(404, f"file not found: {path}")

    max_size = 512 * 1024
    return {"content": data[:max_size], "truncated": len(data) > max_size}

## 3. P1：worker 谎报成功（D13）—— 完整性校验前移到"任务完成"层

### 问题（真机）
GATE-A 测试里 dev-worker 的 `write_file` 被 hook BLOCK，但 worker summary 仍声称"Created src/gate_a.py... Verified"——文件根本不存在。当前完整性校验在 **release 前**（orchestrator `min_release_ok` + policy QA gate）+ **事后告警**（monitor `check_dev_commits`）。报告建议：作为 **dev-worker complete 的前置**。

### 现实约束
`kanban_complete` 由 agent 在容器内调，平台无法在 Hermes 内部拦截调用。但能在**平台侧**对"已 done 的 dev 卡"做独立核验并**纠偏**。

### 改法（两手，都不依赖改 Hermes）

**(A) SOUL 强化（已部分做，再收紧）** `SOUL.dev-worker.md` 增加：
```text
完成判定：只有当 `git -C <你的worktree> status --short` 干净、且 `git log -1` 显示你本次的提交时，
才可调用 kanban_complete。metadata.commit_sha 必须是真实存在的提交哈希。
若你的写操作被策略闸门 block，禁止声称完成——改为 kanban_block 并说明缺哪个 approved/allowed_paths。
```

**(B) watchdog 平台侧纠偏（新增检查）** `platform/watchdog.sh`：对"状态 done 但 worktree 无对应提交 / 该卡 allowed_paths 列出的文件无改动"的 dev 卡，自动 **reopen**（或建复验卡），把"谎报"挡回开发态而非流到 release。给出实现骨架：
```bash
# watchdog.sh：dev 卡 done 但产物没落地 → 建复验卡（幂等）交回 dev-lead
echo "$all_json" \
 | jq -r '.[] | select((.assignee//"")|startswith("dev-worker"))
                | select(.status=="done")
                | "\(.id)\t\(.workspace // "")"' \
 | while IFS=$'\t' read -r tid wt; do
     [ -z "$tid" ] && continue
     # worktree 路径存在但无该卡的提交（按提交信息含 task_id 粗判）→ 疑似谎报
     if [ -n "$wt" ] && [ -d "$wt" ] \
        && ! git -C "$wt" log --oneline 2>/dev/null | grep -q "${tid}"; then
       hermes kanban --board "$pid" create "复验：${tid} 声称完成但 worktree 无对应提交" \
         --assignee dev-lead --idempotency-key "verify-noartifact-${tid}" \
         --body "dev 卡 ${tid} 状态 done 但其 worktree 无 ${tid} 的提交，疑似谎报，请复核。" 2>/dev/null || true
     fi
   done
```
> 这条与现有 monitor `check_dev_commits`（项目级粗粒度告警）互补：watchdog 做**单卡级纠偏动作**，monitor 做**项目级观测告警**。

## 4. glm_monitor.sh / glm_cycle.sh 的定位（部分借鉴，不整体并入）

这两个脚本做三件事：① 清过期 `.provider_pause`；② 跑 `orchestrator tick` + `watchdog`；③ 看板快照投递到聊天。

- **① 已并入** monitor（§1-C 的 `_clear_expired_pause`）。
- **② 是重复**：本平台已用三个 systemd timer（orchestrator/watchdog/monitor 各自周期）。再加一个每 30 分钟也跑 orchestrator+watchdog 的 cron，会与 timer **并发 double-tick**——同一时刻两个 `orchestrator tick` / 两个 `watchdog` 可能并发建卡/续跑，破坏幂等假设。**不并入**。
- **③ 快照投递到聊天**是有用的**运维播报**，但它硬编码 `HERMES_BIN="$HOME/.hermes/hermes-agent/venv/bin/hermes"` 和 cron-to-chat，可移植性差。

### 结论与改法
- **不把 glm 脚本入库**。把唯一缺的能力（清过期 pause）并进 monitor（§1-C）。
- 若你要"定时把看板快照推到聊天/webhook"，正确做法是给 **monitor 增加一个 `--snapshot` 模式**或独立的轻量 `platform/report_snapshot.sh`（用 `command -v hermes` 而非硬编码路径，PLATFORM_DATA_ROOT 可配），由你已有的 monitor timer 顺带产出，避免再起一个会 double-tick 的 cron。
- **文档明确**：`03` 加一句"自动化只走部署装的三个 systemd timer；不要再额外 cron 跑 orchestrator/watchdog，会重复 tick"。
