## 2. 真正要改的（改哪里、怎么改）

### P0：异常检测改按 `status`（不只 last_event）——D14
**问题**：`watchdog.sh:51` 和 `monitor.sh:50 check_stuck` 只按 `last_event` 筛异常；该字段在 `kanban list --json` 里可能为空 → 续跑与告警双双失效（真机实测 0 匹配）。`status` 字段（blocked/failed）才是可靠信号。

**改 `platform/monitor.sh` check_stuck（纯告警，安全）**：
```bash
n=$(HERMES_HOME="${home}" hermes kanban --board "${pid}" list --json 2>/dev/null \
    | jq '[.[] | select(.last_event=="gave_up" or .last_event=="timed_out"
                        or .last_event=="stale" or .last_event=="protocol_violation"
                        or .status=="blocked" or .status=="failed")] | length' 2>/dev/null)
```
→ 卡死/失败任务无论 last_event 是否为空都能被看见、告警。

**改 `platform/watchdog.sh`（分类处理，不盲目续跑）**：把"只认 last_event"扩成"先按 status 捞 blocked/failed，再按原因分类"——这正是不该盲目 unblock（D18 教训）的体现：
```bash
echo "$all_json" \
 | jq -r '.[] | select(.last_event=="gave_up" or .last_event=="timed_out"
                      or .last_event=="stale" or .last_event=="protocol_violation"
                      or .status=="blocked" or .status=="failed")
          | "\(.id)\t\(.assignee // "dev-worker-1")\t\(.workspace // "")\t\([.block_reason,.reason,.blocked_reason]|map(.//"")|join(" ")|ascii_downcase)"' \
 | while IFS=$'\t' read -r tid assignee workspace reason; do
     case "$reason" in
       *environment*|*mount*|*docker*|*wrong\ workspace*)
         # 环境/挂载错误：续跑必再失败 → 不续跑，建 CRIT 隔离卡（幂等）交人工/触发 docker guard
         _mkcard "$pid" "环境/挂载异常需排查：${tid}（${reason}）" change-guardian "wd-env-${tid}" ;;
       *review*) : ;;  # review-required 由下方既有 AUTOCODE_AUTO_APPROVE_REVIEW 分支处理
       *) _continue_task "$pid" "$tid" "$assignee" "$workspace" ;;  # 其余按原有续跑（继承 allowed_paths）
     esac
   done
```
（把现有续跑主体抽成 `_continue_task`，环境类走 `_mkcard` 建排查卡而非续跑。）

**配套**：`tests/test_static_contracts.py` 断言 watchdog/monitor 含 `.status=="blocked"`；`03` 监测表把 check_stuck 口径更新为"按 status+last_event"。

### P1：worker 的 Docker 后端可靠性——D26/D29（保 docker，不改 local）
**不采纳**报告的 `backend: local`——那会让 dev/qa/release 在**宿主**跑代码，彻底破坏"执行必须在非 root Docker 沙箱"的隔离红线。根因是 docker 组没进 gateway 的 systemd 用户服务上下文。三件事：

1. **部署期早失败 + 明确指引**（`scripts/01-deploy-platform.sh`）：装完 gateway 单元后，在**用户服务上下文**探测 docker 可用性，不可用就 CRIT 提示（而非等 worker 跑到一半才 blocked）：
   ```bash
   systemctl --user show-environment >/dev/null 2>&1
   if ! systemd-run --user --scope -q docker info >/dev/null 2>&1; then
     echo "❌ 用户 systemd 服务上下文访问不了 Docker——worker 会全部 blocked。" >&2
     echo "   修复：重新登录，或 'sudo systemctl restart user@\$(id -u).service' 让用户管理器拿到 docker 组，再重跑部署。" >&2
   fi
   ```
2. **运行期可观测**（`monitor.sh` 新增 `check_worker_backend`）：检测"dev-worker/qa/release 卡 status=blocked 且原因含 docker/environment"→ CRIT，直指 docker 访问问题（依赖 P0 的 status 检测）。
3. **文档**（`03`/`00-host-setup`）：把"docker 组必须对 user systemd 生效（重登或 restart user@uid）"从排障提示**升级为 Step 1 的硬前置**，并解释：worker 由 gateway 用户服务派生，用户管理器没拿到 docker 组 → 整条流水线停摆。
> 注：`backend: docker` 本身是对的、必须保留；这是**环境配置**问题，不是改 backend。

## 4. 不采纳清单（附理由，避免削弱设计）
