### 判定为【不实/不采纳】及理由

| 来源 | 建议 | 不采纳理由 |

### 报告间互相矛盾的仲裁

- **dev-lead 处置预存代码**：报告 A（7.1）让 dev-lead"直接 commit 进 QA"，报告 B（P0-2）指出 dev-lead 根本没有 terminal。**B 对 A 错**（工具配置确凿）。但 B 的"worktree 里做 baseline"也有物理问题——主 ws 的**未提交**文件在新 worktree 里不可见，故最终方案是 dev-worker 在**主 workspace** 做 baseline（policy 的 taskless 兜底/allowed_paths 均可覆盖，且有 policy_fallback 可观测）。
- **B 的 reports/dev/status.json 新证据契约**：未采纳——引入一个 QA/policy/SOUL 三方都要新学的文件契约，而"看板 dev-worker 卡 done + git 真实提交"两个既有信号已足够表达同一证据链（提交判据与 release 硬闸同口径）。少一个契约就少一类漂移。

## 三、流程层面的最终口径（更新后）

```
需求确认：用户↔CEO 对话 → CEO 给定版四块 → 用户点击确认
         → 平台校验（不合法→409/422 带原因，绝不写占位）→ 落盘 → 产品委员会
预存代码：dev-lead 建 baseline-validation 卡（dev-worker 在主 ws 核对/测试/commit）
         → dev-worker 卡 done + git 真实提交 → QA（编排器同判据自愈兜底）
异常分类：agent 可修 → 续跑/repair 卡；1305 → pause 自愈；
         环境/宿主问题 → operator_required（UI 红条+审计，不空转）
完成判定：可交付(is_done) 与 自然无人值守完成(is_unattended_done) 分开度量；
         complete 时归档遗留卡，看板即最终真实状态
```
