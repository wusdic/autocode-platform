## 0. 结论先行

报告证明：**这套多角色编排在真实 Hermes v0.16 上是能跑通的**，设计方向成立。阻碍它"开箱即用"的不是架构，而是一批**Hermes v0.16 的命令/配置格式细节**——它们在静态分析里看不出来，只有真机跑才暴露。本方案 = 把这些细节一次性对齐。

最关键的三类：
1. **配置值格式**：Hermes v0.16 的 `toolsets`/`disabled_toolsets`/`terminal.docker_volumes` 期望 **JSON 数组**，而仓库写的是逗号字符串/bare string → 权限隔离与 Docker 挂载全部失效。
2. **第二层安全形同虚设**：插件被复制进目录但**从未 `hermes plugins enable policy`** → 设计闸门整条流水线缺席（靠第一层 toolset 兜住才没出事）。
3. **CLI flag 名不符**：`swarm --workers` 应为 `--worker`（单数，可重复，`PROFILE:TITLE` 格式）→ confirm-plan 必失败。

## 1. 🔴 P0 — Hermes v0.16 CLI / 配置格式对齐（必改）

### 1.1 `control_plane.py` · swarm flag：`--workers` → `--worker`（NEW-E）⬜
**位置**：`HermesGateway.swarm()`。
**现状**：
```python
cmd = ["hermes","kanban","--board",project.project_id,"swarm",goal,
       "--workers", ",".join(workers),     # ❌ v0.16 无 --workers
       "--verifier",verifier,"--synthesizer",synthesizer]
```
**改成**（`--worker` 单数、可重复、`PROFILE:TITLE` 格式）：
```python
cmd = ["hermes","kanban","--board",project.project_id,"swarm",goal]
for w in workers:
    cmd += ["--worker", f"{w}:{w}"]
cmd += ["--verifier", verifier, "--synthesizer", synthesizer]
```
**为什么**：v0.16 实际签名 `hermes kanban swarm <goal> --worker PROFILE:TITLE[:SKILLS] --verifier V --synthesizer S`。不改 → Step 6 confirm-plan 启动产品委员会 swarm 直接报错。
**测试**：把 `test_confirm_plan_creates_product_swarm` 的 FakeGateway.swarm 断言改为接收 worker 列表后仍成立（已有）。

### 1.2 `launch_project.sh` · 配置值改 JSON 数组（NEW-K / NEW-M / NEW-L）⬜
Hermes v0.16 的列表型配置用 `config set` 写入时需 **JSON 数组字符串**；逗号字符串会被当作单个无效值。

**(a) `terminal.docker_volumes`**（NEW-M，dev/qa 4 角色）：
```diff
- hermes -p "$r" config set terminal.docker_volumes "${WORKSPACE}:${WORKSPACE}"
+ hermes -p "$r" config set terminal.docker_volumes "[\"${WORKSPACE}:${WORKSPACE}\"]"
```
不改 → 4 个研发角色 Docker 容器启动即 `ValueError: Invalid value for TERMINAL_DOCKER_VOLUMES (expected valid JSON)`，无法跑代码（报告里 dev-worker-1 因此中断 9 小时）。

**(b) `toolsets` / `agent.disabled_toolsets`**（NEW-K / NEW-L）：
- 关键认知：v0.16 的 `toolsets` 是**附加列表**（加插件工具集），**内置工具（terminal/file/code_execution）始终可用**；要禁用必须用 `agent.disabled_toolsets`。所以**真正的第一层权限 = `disabled_toolsets`**，且值要 JSON 数组。
- 且危险工具的真实名是 **`code_execution`**（不是只有 terminal）——CEO 当时就是用 `code_execution` 直接写了 158 行代码绕过流程（NEW-L）。

```diff
# CEO（只沟通）
- hermes -p ceo config set toolsets "clarify,kanban,memory,messaging"
- hermes -p ceo config set agent.disabled_toolsets "terminal,file,patch,web,browser"
+ hermes -p ceo config set toolsets '["clarify","kanban","memory","messaging"]'
+ hermes -p ceo config set agent.disabled_toolsets '["code_execution","terminal","file"]'

# no-code 设计角色（可写 design/，不可执行/改码）
- hermes -p "$r" config set agent.disabled_toolsets "terminal,patch"
+ hermes -p "$r" config set agent.disabled_toolsets '["code_execution","terminal"]'
```
> 注：仅用**确认存在**的工具集名（`code_execution`/`terminal`/`file`/`clarify`/`kanban`/`memory`/`messaging`）。`patch`/`web`/`browser` 是否为合法 toolset 名未经真机确认，先不写进去以免 `set -e` 中止；第二层 policy 插件会兜底拦 patch。

### 1.3 `launch_project.sh` · gateway 启动：`start` → `run`（NEW-I）⬜
**位置**：生成 systemd unit 的 `ExecStart`。
```diff
- ExecStart=$(command -v hermes) -p ceo gateway start
+ ExecStart=$(command -v hermes) -p ceo gateway run
```
**为什么**：`gateway start` 要求已 `gateway install` 的机器级 systemd 服务；项目独立 HERMES_HOME 下并未安装 → 无限重启失败 `Gateway service is not installed`。`gateway run` 是前台模式，正好交给我们自己的 systemd unit（`Type=simple`）管理。**这同时解答了之前一直存疑的"Type=simple 是否会被后台化"——用 `run` 就是前台，Type=simple 正确。**

### 1.4 `launch_project.sh` · 启用插件：补 `hermes plugins enable policy`（NEW-O）⬜
**位置**：第 5 步复制 `plugin.py` 到 `${HERMES_HOME}/plugins/policy/` 之后。
```bash
# 复制 plugin.py / plugin.yaml 之后，新增：
HERMES_HOME="${HERMES_HOME}" hermes plugins enable policy
```
**为什么**：**插件只复制不 enable，Hermes 根本不加载它**——报告确认整条开发流水线期间第二层设计闸门完全缺席（全靠第一层 toolset 撑着）。这是"双层硬拦"里第二层从未生效的根因。不改 → 安全模型只剩一层。

### 1.5 `monitor.sh` · `config get` → `config show`（NEW-F）⬜
**位置**：`check_first_layer()`。
```diff
- dis=$(HERMES_HOME="${home}" hermes -p ceo config get agent.disabled_toolsets 2>/dev/null || echo "")
+ dis=$(HERMES_HOME="${home}" hermes -p ceo config show 2>/dev/null | grep -A3 disabled_toolsets || echo "")
- if ! printf '%s' "${dis}" | grep -q "terminal"; then
+ if ! printf '%s' "${dis}" | grep -q "code_execution"; then
```
**为什么**：v0.16 `hermes config` 子命令只有 `show/edit/set/path/env-path/check/migrate`，**没有 `get`** → 现有健康检查永远拿到空串、误报。并把判据从 `terminal` 改为 `code_execution`（真正该禁的工具）。

## 2. 🟠 P1 — 部署稳定性

### 2.1 控制平面做成 systemd 服务（NEW-P / 26.3 / 26.4 / NEW-J）⬜
**现状**：`scripts/01-deploy-platform.sh` 末尾只打印一条 `nohup uvicorn ...`，session 结束/清理 /tmp 即丢；且 systemd 用户服务默认 PATH 不含 hermes venv → subprocess 找不到 `hermes`（创建项目 500）；token 重启后回落 `change-me` 不匹配。
**改**：仓库新增 `platform/autocode-control-plane.service.template`，部署脚本据此生成：
```ini
[Unit]
Description=Autocode control plane
After=network.target
[Service]
Type=simple
Environment=PLATFORM_TOKEN=__TOKEN__
Environment=PLATFORM_DATA_ROOT=/data/projects
Environment=XDG_RUNTIME_DIR=/run/user/__UID__
Environment=PATH=__HERMES_VENV_BIN__:/usr/local/bin:/usr/bin:/bin
ExecStart=__PLATFORM_HOME__/venv/bin/uvicorn control_plane:app --app-dir __PLATFORM_HOME__ --host 127.0.0.1 --port 9000
Restart=always
RestartSec=3
[Install]
WantedBy=default.target
```
覆盖 NEW-P（持久）、26.4（PATH 含 hermes）、NEW-J（XDG_RUNTIME_DIR）、26.3（token 固定在 Environment）。

### 2.2 `control_plane.py` · PLATFORM_TOKEN 文件 fallback（NEW-H）⬜
**为什么**：Hermes 的密钥脱敏会阻止把 token 写进脚本文件传递。
**改** `Settings.token` 默认值：env → `~/platform/.platform_token` 文件 → `change-me`。
```python
def _default_token() -> str:
    env = os.environ.get("PLATFORM_TOKEN")
    if env:
        return env
    p = os.path.expanduser("~/platform/.platform_token")
    if os.path.exists(p):
        return open(p).read().strip()
    return "change-me"
# token: str = field(default_factory=_default_token)
```

### 2.3 Docker 产物属主（26.1）⬜
**现象**：Docker 沙箱内以 root 写文件，宿主用户读不了（`PermissionError`）。
**改**：两选一——
- `launch_project.sh` 给 dev/qa 设 `terminal.docker_user` 映射当前 UID（若 v0.16 支持该键，需 `--help` 确认）；或
- 由 `monitor.sh`/cron 周期性 `chown -R $USER /data/projects/*/workspace`（兜底，已写进监控建议）。
本方案推荐：在监控里加 chown 兜底（确定可行），并把 docker_user 列为待验证项。

## 3. 🟡 P2 — 模型配置外部化（§12，强烈建议）

### 3.1 现状问题
17 个角色的 `model/provider/base_url` **硬编码在 `launch_project.sh` 的 `create_role` 调用里**，且原始 `create_role` 只设 `model.default`、**不设 `model.provider`/`model.base_url`** → 无法跨供应商路由，占位模型名直接报错。

### 3.2 改法（落地版）
**(a) `create_role` 增参**：
```bash
create_role () {  # name model provider base_url desc
  local name="$1" model="$2" provider="$3" base_url="$4" desc="$5"
  hermes profile create "${name}" --description "${desc}" || true
  hermes -p "${name}" config set model.default "${model}"
  hermes -p "${name}" config set model.provider "${provider}"
  hermes -p "${name}" config set model.base_url "${base_url}"
  hermes -p "${name}" config set agent.max_turns 200
  hermes -p "${name}" config set terminal.cwd "${WORKSPACE}"
}
```
**(b) 把 17 行映射抽到 `platform-base/model-config.tsv`**（单一编辑点，TSV：`role⇥model⇥provider⇥base_url⇥description`），`launch_project.sh` 循环读取；文件缺失时回落内置默认。供应商 key 仍只放 `~/.hermes/.env`。
**好处**：改模型只改 TSV、可按项目覆盖（`launch_project.sh <pid> <port> [model-config.tsv]`）、配置集中。
