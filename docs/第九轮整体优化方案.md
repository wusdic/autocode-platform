# 第九轮整体优化方案：运行时可靠性闭环（已实现）

> 输入：第七轮真机报告（跑到 release，D14–D31）+ 两份评审（GPT 终版 + 逐行核对版）。基线 `f7180ae`→ 本轮 `main`。
> 主题：从"权限/隔离修补"转向**运行时可靠性闭环**——让平台真正能无人值守跑完，而非靠 operator 手搓。
> 原则：统筹设计、不头疼医头、生产级；削弱安全/隔离的做法一律重写或不采纳。

## 0. 两份评审与我的结论一致
真机首次走完全流程（42/42 测试 + 真实 manifest），验证架构可用。两份评审 + 我的分析都指向同两个**仓库未修的真 P0**：**D26 worker Docker 后端不可用**、**D14 watchdog 只认空字段 last_event**。逐条核验后本轮系统性修复。

## 1. 已实现（改哪里、怎么改）

### P0-A — worker Docker 后端可靠（D26/D29），坚持隔离不降级
**根因**：gateway 是 systemd user service，派生的 worker 子进程默认**不继承用户的 docker 组**（即便用户在 docker 组）→ worker 调 docker daemon 失败、dev/qa/release 全卡死，只能 operator 手搓。
- **根治（采纳评审#2 的关键洞察）**：gateway 单元 + 控制平面单元加 `SupplementaryGroups=docker`（仅当宿主有 docker 组时写）——worker 子进程天然拿到组权限，**根本不必降级 local**。`launch_project.sh` / `01-deploy-platform.sh`。
- **建项目期早失败**：`AUTOCODE_EXECUTOR_BACKEND`（auto|docker|local）+ `AUTOCODE_ALLOW_LOCAL_EXECUTOR`（默认 0）。auto 时探测运行上下文能否 `docker version`，不行就**拒绝建项目**并给"重登/restart user@uid"指引；**绝不静默回退 local**（local 在宿主跑代码=破坏隔离红线，只有显式 `AUTOCODE_ALLOW_LOCAL_EXECUTOR=1` 才降级，并写 `.autocode/executor_backend.json{degraded:true}`）。
- **派发前校验（D29）**：`_verify_worker_profiles` 在启动 gateway 前读回每个 worker config，校验 toolsets 含 terminal/file/kanban、disabled_toolsets 是列表且含 execute_code、backend 与选定一致——把"worker 缺工具/后端不对"从"跑到 dev 才卡死"提前到"建项目就失败"。
> **不采纳**报告的"改 backend: local"——那是把安全红线让给可用性。正确是修 docker 访问环境。

### P0-B — 异常检测改按 status，不靠空的 last_event（D14）
**根因**：真机 `kanban list --json` 的 `last_event` 字段为空（事件只进 task_events 表），`watchdog.sh`/`monitor.sh` 只按它筛 → 匹配 0 条、**续跑与卡死告警双双静默失效**（无人值守在限流场景彻底破灭）。
- `watchdog.sh`/`monitor.sh check_stuck` 改为按可靠的 **`status`**（blocked/failed/…）捞异常任务。
- watchdog **按 reason 分类**（不盲目 unblock，呼应 D18 教训）：余额不足/billing → 不续跑；环境/挂载错误 → 建 change-guardian 排查卡而非续跑；其余 → 原有续跑（继承 allowed_paths）。

### P0-C — 余额耗尽(1113) 永久故障：不重试（D13/D19）
1113（余额不足）与 1305（临时过载）本质不同：前者重试无意义、需充值。
- `monitor.sh` 检测 1113 → 除 CRIT 外写 `.provider_billing_dead`（区别于 1305 写 `.provider_pause`）。
- `watchdog.sh` 见 billing_dead → 跳过本轮续跑；`orchestrator.provider_paused` 把 billing_dead 视同暂停 → 不起新 swarm。充值后人工 `rm .provider_billing_dead` 恢复。
- `check-models.sh` 解析 429 body 区分 1113（硬错误 fail-fast）/1305（警告），并提示"检测的是项目 gateway key，不是你聊天会话的 key"（消除报告 D23 的误解）。

### P1-A — 配置不能被静默写坏（D27）
worker 校验（P0-A 的 `_verify_worker_profiles`）用 PyYAML **读回类型校验**：toolsets/disabled_toolsets 必须是 YAML 列表（被写成字符串就 fail），杜绝报告里"echo 把列表写成字符串 → Hermes 静默 fallback"的隐患。

### P1-B — 状态机 direct-to-QA 路径（D30）
真机 demo1 串行交付、无 dev-worker fan-out 卡 → `_all_dev_done` 永远 False、卡死在 dev→qa。
- 新增 `_dev_complete`：dev-worker 卡全 done（正常）**或** dev-lead 卡 done 且**已有真实源码落地**（direct-to-QA）→ 进 QA。靠 `expected_files_present` 防"空手放行"；release 前仍走 `min_release_ok` 完整性硬闸。
- `SOUL.dev-lead`：发现 workspace 已有未跟踪代码时的策略——优先 fan-out 核对；确已覆盖则 commit 基线 + 完成自己的卡走 direct-to-QA；**绝不**停在"给用户 A/B/C 选项 block 等人工"（那不是设计中的人工介入点，是 D16 越权产物的副作用）。

## 2. 判定为"已覆盖/他们的问题/不采纳"
- **D16 arch-synthesizer 越权写码**：policy 已挡（no-code 角色 write_file 只放行 design/，且禁 terminal/execute_code）。报告里能越权是 **D27 配置损坏**（disabled_toolsets 被写成字符串而失效）。本轮的 `_verify_worker_profiles`（类型校验）正是堵这个根因。**不在 release 闸加"主 workspace 审计"**——评审#2 建议的这条会误杀**合法串行项目**（代码直接提交到 main，如 demo1 的 todo.py），scope_guard 无法区分"dev-worker 合法写 main"与"越权"，会把正常串行交付拦在 release 外，属回归，故不采纳。D16 的治本是 hook 生效 + 配置校验，不是事后审计。
- **backend: local 默认**、**反复手动 unblock**、**approved 宽松匹配**、**chmod 666 docker.sock**：均削弱安全/设计，不采纳（与评审一致）。
- **D29 Hermes #22924/#22922**：上游问题；平台侧已显式配 toolsets + 派发前校验兜底。
- **D28 session 工具集锁定**：Hermes 行为；本轮"建项目期校验 + 早失败"已大幅降低其影响（配置在 spawn 前就对，无需热更新）。
- **sudo 免密 / 镜像源 / 磁盘 / D31 看板统计噪声**：前几轮已处理或纯统计噪声，低优先。

## 3. 验收
- `make ci`：**178 测试全过** + shellcheck + 文档块校验；实操 e2e **35/35**。
- 新增契约：watchdog/monitor 按 status、billing_dead 三处一致、SupplementaryGroups+backend 选择+worker 校验、direct-to-QA。

> 一句话：这轮把"无人值守"从"测试人员手搓兜底"推进到"真能自己跑完"——**worker 能跑**（SupplementaryGroups 根治 docker 组 + 建项目期早失败，绝不降级 local 牺牲隔离）、**异常能恢复**（按可靠的 status 检测 + 按原因分类，不盲目 unblock）、**永久故障不空转**（1113 billing_dead 不重试）、**状态机无死角**（direct-to-QA 但产物校验把关）、**配置不被写坏**（派发前类型校验）。报告/评审里削弱安全（local 默认、主 ws 审计误杀串行、宽松开闸）的做法一律不采纳。
